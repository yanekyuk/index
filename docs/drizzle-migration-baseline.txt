Drizzle Migration Baseline (Housekeeping)

If `yarn db:migrate` fails with journal/tag errors (e.g. expecting `0000_massive_mongu.sql` that doesn't exist), your local migration journal is out of sync with the files.

This baseline procedure realigns Drizzle without touching existing data.

TL;DR
1) Backup your DB.
2) Ensure the current DB schema matches the code (you may have created tables manually already).
3) Reset Drizzle meta journal to a fresh baseline.

Steps
1. Backup your database.

2. Ensure required extension exists (for UUID defaults):
   CREATE EXTENSION IF NOT EXISTS pgcrypto;

3. If you manually created tables (`index_links`, `integration_items`), keep them — our migrations use IF NOT EXISTS so they won't duplicate.

4. Reset Drizzle meta journal locally:
   - Remove existing meta snapshots (local only):
     rm -rf protocol/drizzle/meta
   - (Optional) Remove generated migration files if they were partially created (leave canonical repo migrations intact)
   - Re-generate snapshots with current schema:
     cd protocol && yarn db:generate

   This creates a new snapshot and a new migration (e.g. 000X_*.sql) representing the current schema. Because our DDL uses IF NOT EXISTS, applying it on a DB that already has the tables will be a no-op.

5. Run migrations cleanly:
   cd protocol && yarn db:migrate

You should see Using 'postgres' driver and zero-op application for existing tables.

Notes
- On shared environments, align on a single set of migration files to avoid divergent journals.
- Prefer IF NOT EXISTS for CREATE statements when baselining on existing DBs.
- For CI/CD or new envs, fresh yarn db:migrate should work with the new baseline.

Troubleshooting
- If drizzle-kit migrate still references old tags, delete and re-create protocol/drizzle/meta and re-run yarn db:generate.
- Ensure protocol/drizzle.config.ts points to the correct schema and output folder.
